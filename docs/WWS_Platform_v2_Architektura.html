Alapelv

Minden AI válasz minősége egyetlen dologtól függ: milyen kontextust kap a modell. A modell ugyanaz — a tool-ok teszik okossá vagy butává.

A platform egy vékony alap kontextust (pulzust) biztosít — a rendszer pillanatnyi állapotának tömör összefoglalóját. Ezen túl az LLM tool-okon keresztül kérdez le részletes adatot, amikor szüksége van rá.

Az irányítás az LLM-é, nem a rendszeré. A rendszer nem próbálja kitalálni, mire lesz szükség — eszközöket ad, és hagyja, hogy a modell döntse el, mit kérdez le. Ez az agentic RAG pattern.

WWS Platform v2 A Háromszintű Architektúra
Miért Három Szint?

Minden LLM alkalmazás végső soron egy kontextus-összeállítási probléma. A modell ugyanaz — ami különbséget tesz: milyen kontextust kap, hogyan van strukturálva, és mikor kapja meg.

A hagyományos megközelítés: a rendszer előre kitalálja, mire lesz szüksége az LLM-nek, összegyűjti, és odaadja. Ez a push modell. A probléma: vagy túl sokat gyűjt (lassú, drága, pontatlan), vagy túl keveset (rossz válasz).

Az agentic RAG pattern megfordítja az irányítást: az LLM dönt, mikor mit kérdez le. A rendszer tool-okat biztosít — az LLM hívja őket igény szerint.

Analógia: A Claude Code nem olvassa be az egész repót a promptba. Read, Grep, Glob tool-jai vannak — és ő dönt, mikor melyiket hívja. Nem a rendszer dönti el előre, melyik fájlokat kell beolvasni. Ugyanezt csináljuk, csak üzleti adatokkal.

WWS Platform v2 Tool Definíciók
A 6+1 Tool

Az LLM számára elérhető tool-ok. Minden tool jól definiált, gyors, parametrizált lekérdezés — ugyanazok az adatok, mint az eredeti modulokban, de tool interfészként.

1 — get_budget_summary
Budget & FP&A

Paraméterek: department?, period?
Visszatér: Budget vs actual, túllépések, osztály bontás
Cache: Redis, 15 perc TTL
Forrás: budget_lines, accounting_entries

Token méret: ~400-600 token
2 — get_invoice_stats
Számla Állapot

Paraméterek: status?, partner?
Visszatér: Státusz megoszlás, lejárt, duplikát gyanús, AI confidence
Cache: Redis, 5 perc TTL
Forrás: invoices tábla aggregációk

Token méret: ~300 token
3 — get_partner_detail
Szállítói Kép

Paraméterek: name_or_id
Visszatér: Partner összeg, fizetési minta, nyitott PO-k, számlák
Cache: Redis, 30 perc TTL
Forrás: partners + invoices join

Token méret: ~300-400 token
4 — get_anomalies
Aktív Riasztások

Paraméterek: severity?, type?
Visszatér: Anomáliák típusonként, érintett partnerek, severity
Cache: Nincs (mindig friss)
Forrás: invoice.anomaly_flags + ai_enrichments

Token méret: ~200-500 token
5 — get_cash_position
Cash & Runway

Paraméterek: scenario?
Visszatér: Cash position, runway szcenáriók, pénzforgalom
Cache: Redis, 15 perc TTL
Forrás: accounting_entries + budget_lines

Token méret: ~400 token
6 — search_history
Releváns Múlt

Paraméterek: topic
Visszatér: Top 3 hasonló korábbi kérdés-válasz, outcome score-ral
Cache: Nincs (kérdésfüggő)
Forrás: Vector DB szemantikus keresés

Token méret: ~300-600 token

+1 Fallback: execute_sql(query) — Ha a fenti tool-ok nem adnak elég részletes adatot, az LLM SQL-t generálhat és futtathat. Guardrails: read-only, timeout, row limit. Ez a V1 text-to-SQL képesség — tool-ként megmarad, de nem az elsődleges út.

Push vs Pull

A kontextus-összeállításnak két modellje van. A v2 architektúra a pull (agentic) modellt követi.

DimenzióPush (előre-összeállított)Pull (agentic, tool-alapú)
Ki dönt? A rendszer — rule-based osztályozás alapján Az LLM — a kérdés megértése alapján
Token pazarlás Magas — előre gyűjt, ami nem biztos, hogy kell Minimális — csak az kerül be, amit az LLM kér
Kapcsolódó kontextus Nehéz — a rendszernek előre kell tudnia, mi függ össze Természetes — az LLM a tool eredmények alapján dönt, kell-e tovább menni
Skálázhatóság Új adattípus = új osztályozási szabály + modul Új adattípus = új tool definíció
Hibamód Rossz osztályozás → rossz kontextus → rossz válasz LLM dönt rosszul → retry / más tool / kérdez vissza

A push modell alapproblémája: a rendszernek a kérdés megértése előtt kell döntenie arról, milyen adatot gyűjt. Ez egy osztályozási feladat — amit ironikus módon egy LLM oldana meg legjobban. De ha már LLM-et használunk az osztályozásra, miért ne hagynánk, hogy ő döntse el, mit kérdez le?

Iparági Referenciák

Az agentic RAG pattern nem elméleti — ez a 2025-ös AI tooling de facto standardja:

  • Claude Code — Read, Grep, Glob, Bash tool-ok. Az LLM dönt, mikor melyiket hívja. Nem a rendszer dönti el előre, melyik fájlokat kell beolvasni.
  • ChatGPT — Web search, code interpreter, file reader. A modell dönt, mikor keres a weben vs. mikor számol.
  • Cursor / Windsurf — Code search, file read, terminal. Az IDE tool-okat ad, az LLM navigálja a kódbázist.
  • Perplexity — Iteratív web keresés. Az első keresés eredménye alapján dönt, kell-e mélyebbre menni.

Mindegyik ugyanazt csinálja: vékony alap kontextus + tool-ok + iteratív mélyülés.

Az Egyetlen Dolog, Amit Push-olunk

Az alap kontextus (pulzus) az egyetlen rész, amit a rendszer „push"-ol — mindig ott van a system promptban. Ez ~300 token: a rendszer pillanatnyi állapotának összefoglalója.

Minden más pull — az LLM hívja, amikor szükséges.

Ez a hibrid megközelítés biztosítja, hogy:

  • Az LLM-nek mindig van rálátása a rendszer állapotára (pulzus)
  • De nem pazarol tokent részletes adatra, amire nem lesz szüksége
  • A részletes adat mindig friss — a tool híváskor kérdezi le, nem cache-ből

Az Alap Kontextus (Pulzus)

Ez mindig a system prompt része. ~300 token, 5 percenként frissül, cache-ből jön. A rendszer pillanatnyi állapotának tömör összefoglalója — ez ad rálátást az LLM-nek.

SYSTEM: Te egy pénzügyi asszisztens vagy. Szabályok: - Magyarul válaszolj - Adatokra hivatkozz, ne tanácsot adj - Ha anomáliát látsz, emeld ki CONTEXT (pulzus): Nyitott számlák: 47 (18.2M Ft) | Lejárt: 12 Budget felhasználás: 78% | Túllépés: IT +12% Anomáliák: 5 aktív (2 kritikus) Cash runway: 8.2 hónap (alap szcenárió) Legutóbbi hasonló kérdés: 2026-01-15 → budget átcsoportosítás → megoldódott TOOLS: - get_budget_summary(department?, period?) - get_invoice_stats(status?, partner?) - get_partner_detail(name_or_id) - get_anomalies(severity?, type?) - get_cash_position(scenario?) - search_history(topic) - execute_sql(query) USER: "miért nőtt az IT költség?"
Példa: Tool Calling Flow

Végigkövetjük, hogyan dolgozza fel az LLM a kérdést: "miért nőtt az IT költség?"

1 — LLM GONDOLKODIK
Alap kontextus elemzése
Látja: „IT +12%" a pulzusban.
Döntés: részletes budget adat kell.
2 — TOOL HÍVÁS
get_budget_summary(department="IT")
Visszatér: IT budget 15M, actual 16.8M,
top tételek: CloudHost 4.2M, AWS 3.8M, Licenses 2.1M
3 — LLM GONDOLKODIK
Iteratív mélyülés
CloudHost kiugró → kell a partner részlet.
4 — TOOL HÍVÁS
get_partner_detail("CloudHost")
Visszatér: 3 számla 48h alatt, összesen 4.2M,
korábbi átlag: 1.4M/hó, +200% növekedés
5 — VÁLASZ
Összefoglaló az egészből
„Az IT +12%-a főként a CloudHost miatt:
3 számla 48h alatt, 4.2M Ft — ez 200%-os
növekedés az átlaghoz képest. Javasolt: ellenőrzés."

A lényeg: A rendszer nem tudta előre, hogy a CloudHost partner adataira is szükség lesz. Az LLM a budget tool eredménye alapján döntött, hogy a partner tool-t is meghívja. Ez az iteratív mélyülés — a 3. szint.

Interakció Napló

Minden AI interakció rögzítésre kerül — beleértve a tool hívásokat is. Ez nem audit log — ez a rendszer tanulási alapja.

{ "id": "uuid", "timestamp": "2026-03-22T14:30:00", "user_id": "uuid", // Input "question": "miért nőtt az IT költség?", "base_context_hash": "abc123", // Tool hívások (sorrendben) "tool_calls": [ {"tool": "get_budget_summary", "params": {"department": "IT"}, "latency_ms": 120}, {"tool": "get_partner_detail", "params": {"name_or_id": "CloudHost"}, "latency_ms": 85} ], // Output "answer_summary": "IT +12% a CloudHost...", "response_time_ms": 2400, "tokens_used": 2100, // Feedback (később kitöltve) "user_reaction": null, "outcome_30d": null, "outcome_90d": null, // Vector "vector_id": "qdrant-point-id" }

A tool hívások naplózása tanulási adat: a rendszer idővel látja, milyen kérdéseknél milyen tool-okat hív az LLM — ebből lehet optimalizálni a tool-okat.

Hogyan Javítja a Válaszokat

Az outcome score visszakerül a vektorba. Amikor a search_history tool hasonló korábbi interakciókat keres, az eredményes válaszok magasabb relevancia-pontot kapnak.

Példa: 3 hónapja valaki kérdezte: "miért nőtt az IT költség?" — a válasz után budget átcsoportosítás történt, és a probléma megoldódott. Outcome: pozitív.

Ma valaki hasonlót kérdez. Az alap kontextus (pulzus) tartalmazza: "Hasonló szituáció 2026-01-ben → budget átcsoportosítás → megoldódott."

Az LLM ezt látva relevánsabb kontextust tud adni — vagy a search_history tool-lal mélyebbre is mehet.

Nem-Triviális Problémák

1. Tool abuse
Az LLM túl sok tool-t hívhat — lassú és drága. Megoldás: tool hívás limit (max 5/interakció), latency monitoring, és a pulzus elég jó kell legyen ahhoz, hogy az egyszerű kérdésekre tool nélkül válaszoljon.

2. Tool design (granularitás)
A tool-ok granularitása kritikus. Túl specifikus = sok hívás kell, lassú. Túl általános = nagy eredmény, token pazarlás. A 6+1 tool bontás a jelenlegi domain alapján optimális, de idővel finomítandó.

3. Cache invalidáció
A tool-ok eredményei cache-eltek. De ha valaki feltölt egy számlát, a get_invoice_stats cache-e érvénytelen. Az event bus megoldja, de a cache invalidáció komplex koreográfia.

4. Outcome tracking valódisága
A 90 napos outcome nehezen mérhető. Honnan tudjuk, hogy a felhasználó döntését az AI válasz befolyásolta? Az ügyfél visszajelzésére kell hagyatkozni — ami ritkán jön.

5. Hidegindítás
Új telepítésnél nincs interakció napló, a search_history tool üres. A pulzus és a többi tool azonnal működik — a tanulási réteg fokozatosan épül.
